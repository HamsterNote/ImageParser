## Context

当前 `ImageParser` 通过 `src/index.ts` 中的 `createOcrText()` 将 OCR 归一化结果写入 `IntermediateText`，再由 `drawDecodedPage()`、序列化测试与 dist 互操作测试消费这些文本字段。现有实现默认依赖 `@hamster-note/types@^0.5.1` 的 `Text` / `IntermediateText` 结构，编码端和解码端都直接读写具体字段，例如：

- 编码阶段直接构造 `content`、`fontSize`、`fontWeight`、`italic`、`width`、`height`、`dir`、`rotate`、`skew`、`isEOL`
- 解码与预览阶段直接消费 `text.content`、`text.dir`、`text.vertical`、`text.rotate`、`text.skew`、`text.width`
- 序列化与 dist 互操作测试默认认为这些字段名称、类型和含义与 `0.5.x` 保持一致

提案已经明确，本次升级到 `@hamster-note/types@0.7.0` 的核心风险不是 OCR 算法本身，而是 `Text` 定义发生了破坏性调整，导致构造、读取、渲染和序列化链路可能在“能编译但语义失真”或“直接类型不兼容”两种失败模式之间摇摆。

因此，本次设计重点不是引入新的 OCR 能力，而是把 `ImageParser` 内部对文本块的依赖边界重新收口：让内部 OCR 归一化结构继续稳定存在，但所有对外 `Text` 结构的写入与读取都通过明确的适配层完成，并以 `0.7.0` 的公开类型定义作为唯一事实来源。

## Goals / Non-Goals

**Goals:**
- 将运行时依赖统一升级到 `@hamster-note/types@0.7.0`，确保源码、测试和 dist 产物围绕同一版本的 `Text` 定义工作
- 收敛 `ImageParser` 内部对 `Text` 字段的写入与读取路径，避免 0.7.0 字段变化散落在多个函数中
- 让 `ImageParser.encode()` 生成的文本块在新版类型下仍可被 `decode()`、序列化流程和预览逻辑稳定消费
- 为方向、旋转、竖排、样式和几何字段定义统一的降级与推导策略，避免字段缺失时行为分叉
- 通过测试锁定“编码 → 序列化 → 反序列化 → 解码 / 互操作”整条链路的兼容契约

**Non-Goals:**
- 不尝试同时兼容 `0.5.x` 与 `0.7.0` 两套对外 `Text` 协议；本次目标是完成向 `0.7.0` 的单向切换
- 不重写 OCR 识别、样式推断或宽度适配算法；这些能力保持现有设计，仅修正它们映射到 `Text` 的方式
- 不修改 `@hamster-note/types` 包内部实现，也不在本仓库复制一份外部类型定义
- 不扩展新的公开 API 或新增自定义序列化协议；继续复用 `IntermediateDocument` 既有机制

## Decisions

### 1. 以 `0.7.0` 公开类型为中心完成单版本切换，而不是维护双版本兼容分支

本次变更将直接把 `package.json` 与锁文件升级到 `@hamster-note/types@0.7.0`，并让源码与测试一起迁移到新版字段语义。`ImageParser` 不在运行时区分“旧版 Text”和“新版 Text”，也不增加双分支判断。

原因是本仓库本质上是 `@hamster-note/types` 的直接消费者，继续维护双版本兼容只会让 `createOcrText()`、`drawDecodedPage()` 和测试断言都膨胀为条件分支集合，反而放大后续维护成本。既然提案已经把这次升级定义为破坏性变更，就应当在设计上把切换做成原子升级：源码、测试、构建产物和文档同时迁移。

备选方案：
- 保留 `0.5.x` 兼容分支并在运行时探测字段：短期看似更稳，但会把类型升级变成长期债务，并让测试矩阵翻倍
- 在本仓库自行声明一层“旧字段别名类型”：会制造与上游定义分叉的第二事实来源，不利于后续继续升级

### 2. 在 `src/index.ts` 内新增文本适配边界，统一负责“写入新版 Text”和“读取新版 Text”

当前代码对文本字段的访问分散在 `createOcrText()`、`getCanvasTextFont()`、`getCanvasTextDirection()`、`getCanvasTextRotation()`、`getCanvasTextSkew()` 等函数中。设计上将保留 `NormalizedOcrTextBlock` 作为内部 OCR 归一化结果，但在它与 `IntermediateText` 之间增加一组聚焦文本适配的 helper，职责分成两层：

- **写入层**：把内部归一化结构映射为 `0.7.0` 可接受的 `IntermediateText` 构造参数
- **读取层**：把 `0.7.0` 的文本实例收敛为解码与预览所需的稳定访问结果，例如方向、旋转角、斜切角、目标宽度、文本内容等

这样做的目的，是把“上游 Text 结构变化”控制在一个很小的边界里。后续如果 `0.7.0` 把某些字段重命名、合并或拆分，修改点应集中在适配 helper，而不是让整个 `decode()` 流程继续到处知道字段细节。

备选方案：
- 直接在现有函数里逐处替换字段名：改动看似最小，但容易遗漏序列化和渲染分支，也不利于后续再次升级
- 彻底把 `IntermediateText` 转成仓库内自定义类型再二次转换：抽象更重，超出当前仓库体量

### 3. 编码端继续以内部归一化几何/样式为事实来源，但通过显式映射规则生成新版文本对象

`runOcr()`、几何分析、样式推断和 `NormalizedOcrTextBlock` 本身不依赖 `@hamster-note/types` 的具体版本，因此设计上保持这些内部结构不变。真正需要调整的是 `createOcrText()` 的输出边界：

- 内容字段由内部 `block.content` 单点映射到新版文本内容表达
- 几何字段继续来自 `block.geometry`，但其落盘位置、字段名与必填约束以 `0.7.0` 为准
- 样式字段如字重、斜体、旋转、斜切、方向等，统一通过适配规则写入，并为缺省值提供稳定兜底
- 对 `0.5.x` 中存在、但 `0.7.0` 可能移除或语义改变的字段，不做机械保留，而是在适配层显式决定“删除、改名、推导还是降级”

核心原则是：内部 OCR 结果表达“识别到了什么”，适配层表达“新版 Text 需要什么”。两者分离后，编码逻辑不会因为上游字段调整被迫重写 OCR 归一化流程。

备选方案：
- 让 `NormalizedOcrTextBlock` 直接镜像 `0.7.0` 文本结构：会把外部类型变化传播到内部 OCR 处理阶段，耦合过深

### 4. 解码与预览只消费标准化访问结果，不再依赖旧字段是否仍然存在

`drawDecodedPage()` 的职责应当是“把一个可解码文本块画出来”，而不是知道 `Text` 每个历史版本的字段差异。因此设计上会把以下访问集中到读取层 helper：

- 文本内容读取
- 文本方向与竖排判定
- 旋转角与斜切角的安全值获取
- 目标宽度与字体相关值获取
- 需要时从新版结构中推导旧行为所需的兼容结果

这意味着 `drawDecodedPage()`、Demo 覆盖层和相关测试应尽量面向“语义访问结果”而不是具体字段名。只要 `0.7.0` 仍能表达这些语义，消费链路就无需知道字段是直接提供还是由适配层推导。

备选方案：
- 保持直接访问 `text.vertical`、`text.dir`、`text.rotate` 等旧字段：一旦上游做字段收敛或嵌套结构调整，解码链路会再次被迫散点修改

### 5. 序列化兼容性以“新版文档可稳定往返”为准，不额外承诺旧版文档自动迁移

提案要求避免升级后出现“构建通过但文档读写失真”的问题。设计上，这里的兼容目标定义为：`ImageParser.encode()` 生成的 `IntermediateDocument` 在 `0.7.0` 下序列化、反序列化后，文本块的关键语义仍可被本仓库的 `decode()` 和互操作测试正确消费。

本次不额外引入面向历史 `0.5.x` 序列化文档的迁移器。若后续需要读取旧文档，应由独立变更明确该迁移范围；否则在这次升级中同时承担“生产新版”与“回放所有旧版”会显著扩大风险面。

备选方案：
- 顺带实现旧版文档到新版文档的自动转换：价值存在，但需要先掌握旧版和新版字段的完整差异矩阵，已超出本次变更边界

### 6. 测试优先锁定链路契约，而不是绑定某个具体字段布局

本次升级最容易出问题的地方，是测试仍然只断言某些旧字段存在，却没有证明整条链路仍然成立。设计上测试会围绕以下契约重组：

- `ImageParser.encode()` 产出的文档在 `@hamster-note/types@0.7.0` 下能正确实例化和通过类型边界
- 文本块在序列化与反序列化后，仍保有解码和预览所需的关键几何 / 样式语义
- `ImageParser.decode()` 可以消费升级后的文本对象并稳定导出结果
- dist 产物对外暴露的类型与运行时实例保持一致，不出现“源码通过、打包产物失配”

测试应尽量校验语义结果和链路稳定性，而不是过度绑定某个临时 helper 或某个字段一定叫旧名字。这样可以在不牺牲升级安全性的前提下，为后续小范围重构留出空间。

## Risks / Trade-offs

- [`0.7.0` 的文本字段调整超出预期，导致单点适配仍需改动多处调用] → 先把写入层和读取层 helper 建起来，再迁移调用点，确保字段差异最多集中在适配边界
- [编码与解码对同一语义的理解不一致，出现“能写不能读”的隐性失真] → 以相同的标准化访问 helper 驱动解码测试与序列化回归，避免双份语义实现
- [测试仍然绑定旧字段名，升级后出现大量无效失败] → 优先把测试断言迁移为“可序列化、可解码、可互操作”的契约式校验，只在必要处断言关键字段
- [未处理的 Demo / 预览字段访问在人工验证阶段才暴露] → 将与预览相关的文本访问也纳入同一读取层，避免 UI 侧继续直接读取旧字段
- [只升级新版生产链路，暂不迁移旧文档读取能力，可能让部分历史样例无法直接回放] → 在本次变更中明确这一取舍，把风险收敛到当前升级目标内，后续若有需要再单独补迁移变更

## Migration Plan

1. 升级 `package.json` 与锁文件中的 `@hamster-note/types` 到 `0.7.0`，确保源码、测试和打包链路指向同一版本。
2. 梳理 `src/index.ts` 中所有直接依赖 `Text` / `IntermediateText` 字段的写入点与读取点，建立统一的文本适配 helper。
3. 调整 `createOcrText()`，把内部 OCR 归一化结果按 `0.7.0` 的字段约束映射为新版文本对象，并定义缺省/降级策略。
4. 调整解码与预览辅助函数，使其只消费标准化访问结果，而不再散点依赖旧字段布局。
5. 更新 `src/__tests__/imageParser.test.ts` 与 `src/__tests__/dist-interop.test.ts`，覆盖编码、序列化往返、解码和外部类型互操作。
6. 验证打包产物与文档示例在新版依赖下保持一致，必要时同步修正文档中的依赖版本说明。

回滚策略：若升级后发现 `0.7.0` 的字段变化需要更大范围的协议调整，可先保留文本适配边界的重构成果，但暂时回退依赖版本与对应调用点修改，确保仓库仍回到 `0.5.x` 的单版本稳定状态，而不是留下半迁移状态。

- 竖排与方向信息在 `0.7.0` 中是继续以显式字段表示，还是需要从更通用的布局结构推导？
- 是否需要同步调整 Demo 中引用的 `@hamster-note/types` 版本与展示字段，避免示例页面仍停留在旧结构语义？
