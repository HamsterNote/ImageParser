## Context

当前图片 OCR 编码链路在 `src/index.ts` 中已经具备两段稳定能力：

- `runOcr()` 调用 `@paddleocr/paddleocr-js`，从 `items[].text` 与 `items[].poly` 归一化出文本块
- `ImageParser.decode()` 与 Demo 预览链路已经能够消费 `IntermediateText` 上的 `fontWeight`、`italic`、`rotate`、`skew` 并正确渲染

但现有 `NormalizedOcrTextBlock` 只保留 `content` 与 `box`，`createOcrText()` 又把样式字段统一写死为 `fontWeight: 400`、`italic: false`、`rotate: 0`、`skew: 0`。这意味着：

- OCR 多边形中已经包含的几何线索没有被利用
- 编码阶段一旦丢失样式，下游 `decode` 与 Demo 即使支持样式渲染，也无法复原
- 后续若要在导出、编辑或可视化环节保留图片文字视觉语义，只能重新做一次样式推断，造成重复成本

本次设计目标是在不改变 `IntermediateDocument` 既有消费方式的前提下，把图片中文字的旋转、倾斜和粗细线索在编码阶段一次性补齐，并通过可预测的降级规则保证输出稳定。

## Goals / Non-Goals

**Goals:**
- 扩展 OCR 归一化结构，使其除文本与包围盒外还携带样式推断结果
- 基于 OCR 多边形和图片像素可观测线索，为 `IntermediateText` 填充 `rotate`、`skew`、`italic`、`fontWeight`
- 保持 `ImageParser.encode()` / `decode()` 对外接口不变，仅增强中间文档内容
- 为信息不足、几何异常、采样失败等场景定义明确降级行为，避免输出抖动
- 用测试与 Demo 预览覆盖常见旋转、斜体、较粗字体场景，确保样式字段能被下游复用

**Non-Goals:**
- 不引入新的 OCR 模型或额外第三方视觉分析依赖
- 不扩展 `IntermediateDocument` / `IntermediateText` 类型定义本身，只使用既有字段承载结果
- 不尝试做精确字体识别、字重品牌映射或复杂排版重建
- 不重做 Demo 交互，仅保证它能展示新增样式结果

## Decisions

### 1. 扩展内部归一化结构，先沉淀样式线索再映射到中间文档

`NormalizedOcrTextBlock` 将从仅包含 `content` 与 `box`，扩展为同时携带样式提示，例如 `rotate`、`skew`、`italic`、`fontWeight`。`normalizeOcrResult()` 负责把第三方 OCR 输出与图片分析结果统一收敛为内部块结构，`createOcrText()` 只负责把这些值稳定写入 `IntermediateText`。

这样做可以把“样式推断”和“文档映射”拆开：前者聚焦识别逻辑，后者聚焦数据落盘，避免未来继续在 `createOcrText()` 中堆积判断分支。

备选方案：
- 直接在 `createOcrText()` 内联所有样式推断：改动最少，但会让构造函数承担几何分析和像素采样职责，后续难测试
- 直接把第三方 `item` 原样透传到 `IntermediateText` 构造过程：耦合外部库结构，未来 OCR 输出字段变化时波及主流程

### 2. 旋转与倾斜优先基于 OCR 多边形几何特征推断

`item.poly` 已经提供文本区域四边形。设计上优先利用几何信息推断文字方向：

- 使用上边或下边的主方向角度作为 `rotate` 的基础值
- 通过上下边是否平行、左右边的水平偏移差，估算局部 `skew`
- 当倾斜角超过阈值且与整体旋转可区分时，将 `italic` 标记为 `true`
- 对点数不足、边长过短、角度异常抖动的块直接回退到 `rotate: 0`、`skew: 0`、`italic: false`

原因是几何线索已经来自 OCR 检测框，成本低、稳定性高，并且天然与当前 `decode` 的旋转/斜切渲染方式兼容。

备选方案：
- 完全依赖像素级霍夫变换或重新做边缘检测：理论上更精细，但实现复杂度高，与当前仓库定位不符
- 只记录 `rotate` 不记录 `skew`/`italic`：实现更简单，但无法满足“倾斜感和斜体语义可复用”的目标

### 3. 字重采用轻量像素密度启发式估算，并收敛到有限档位

OCR 多边形无法直接提供字重，因此设计上增加一个轻量图片采样步骤：

- 根据 `box` 或 `poly` 从原图裁出文本区域到离屏 canvas
- 对采样区域做灰度/亮度归一化，统计深色像素占比与笔画填充率
- 结合文本区域高度、文本长度与覆盖率，将结果映射到有限字重档位，例如 `400` / `500` / `600` / `700`
- 对采样失败、对比度过低或结果不稳定的情况统一回退到 `400`

字重不追求排版级精度，而是提供“正常 / 稍粗 / 较粗”的稳定区分，让下游渲染至少不再全部落到默认值。

备选方案：
- 永远固定 `fontWeight: 400`：实现最稳，但无法满足本次目标
- 引入独立字体视觉分类库：精度可能更高，但超出当前项目体量，也会增加浏览器端依赖和性能负担

### 4. 中间文档继续复用现有字段，不新增下游协议

最终样式结果仍然通过 `IntermediateText` 现有字段落盘：

- `rotate` 保存旋转角
- `skew` 保存几何倾斜角
- `italic` 作为斜体语义开关
- `fontWeight` 保存收敛后的字重档位

`decode`、Canvas 渲染与 Demo 无需理解新的专有结构，只需要继续消费已有字段即可。这保证了改动集中在编码端，消费端只做兼容验证而非协议升级。

备选方案：
- 在中间文档中新增 `styleHints` 扩展字段：信息更完整，但会放大类型与下游兼容面，不符合本次“增强而不改协议”的目标

### 5. 测试与 Demo 以“样式字段可复用”为验证核心

测试策略不验证启发式实现细节本身，而验证稳定输出契约：

- `ImageParser.encode()` 在给定旋转/倾斜检测框与可区分粗细样本时，产出的 `IntermediateText` 带有非默认样式值
- 信息不足时回退到默认样式，且不会抛出额外错误
- `ImageParser.decode()` 与 Demo 预览继续使用同一批样式字段完成旋转、斜切、字重渲染

这样做能让测试锁定“中间文档是否可复用”，而不是绑定某个具体阈值公式，降低未来调参成本。

备选方案：
- 只测 `encode()` 输出快照：能发现字段变化，但难以证明 `decode` / Demo 真正复用了这些值
- 只做 Demo 人工验收：成本高且缺乏稳定回归保障

## Risks / Trade-offs

- [几何阈值设置不当，导致普通水平文本被误判为斜体] → 使用有限阈值区间并在测试中覆盖接近边界的样本，优先保守判定
- [像素密度估算受背景图案或低对比度影响，字重误判] → 将字重限制为少量档位，并在置信不足时统一降级到 `400`
- [样式推断逻辑增加 `encode()` 耗时] → 将分析限定在单个文本块的局部区域，避免全图扫描；必要时保留后续按需优化空间
- [OCR 多边形顺序或形状不稳定，导致角度计算异常] → 在归一化阶段先做点位合法性校验与排序兜底，异常时直接回退默认样式
- [Demo 与测试样本无法稳定体现粗细差异] → 使用受控 fixture 和固定 mock 图片区域，避免依赖不可控在线资源

## Migration Plan

1. 扩展 `NormalizedOcrTextBlock` 与相关辅助类型，为样式字段预留明确结构。
2. 在 OCR 归一化流程中增加几何分析辅助函数，基于 `poly` 推断 `rotate`、`skew`、`italic`。
3. 增加离屏 canvas 的局部采样逻辑，输出收敛后的 `fontWeight`，并定义统一降级规则。
4. 更新 `createOcrText()`，将内部样式提示稳定映射到 `IntermediateText`。
5. 调整 `src/__tests__/imageParser.test.ts` 中的 mock 与断言，覆盖旋转文本、斜体文本、较粗字体和降级场景。
6. 验证 `decode` 与 `demo/demo.js` 的覆盖层预览继续消费同一份中间文档，不需要额外协议转换。

回滚策略：若样式推断造成明显误判或性能回退，可保留新增辅助结构但暂时让其统一输出默认样式，或直接回退到仅保留文本与包围盒的既有路径，不影响基础 OCR 能力。

## Open Questions

- 字重档位是否需要严格限定为 `400/700` 两档，还是保留 `500/600` 作为过渡档位更利于 Demo 观察？
- `italic` 是否应当仅由 `skew` 阈值推出，还是在某些小角度场景下允许 `italic: true` 且 `skew: 0` 的保守表达？
- 对竖排或近乎竖排文本，本次是否只保留 `rotate` 结果，暂不尝试推断 `vertical` / `TextDir.TTB`？
