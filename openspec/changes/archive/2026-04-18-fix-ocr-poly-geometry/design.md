## Context

当前 OCR 编码链路已经能消费 `items[].poly`，但几何语义仍然分散在两套逻辑中：

- `src/index.ts` 里的 `toBoundingBoxFromPoints()` 先把 `poly` 压成轴对齐包围盒，再把 `width`、`height`、`x`、`y` 写入 `IntermediateText`
- `inferGeometryStyleHints()` 再单独从四边形推导 `rotate` / `skew`，导致尺寸与角度来源不一致
- `demo/demo.js` 的红框预览直接基于中间文档里的 `x`、`y`、`width`、`height` 叠加旋转/斜切绘制，因此一旦编码阶段的尺寸语义不对，预览就无法稳定贴合真实文字区域

这会带来两个直接问题：

1. OCR `poly` 明明已经给出“第 0 个点为左上角、后续按顺时针排列”的几何信息，但当前实现把它退化成轴对齐框，宽高会被倾斜或旋转放大。
2. 编码、`decode` 与 Demo 预览使用的几何基准并不完全一致，同一文本块在中间文档里看起来“有角度”，但尺寸和位置仍然是旧语义，导致红框与真实文字区域错位。

本次设计目标是把 `poly` 升级为 OCR 文本块的唯一几何真源，统一编码输出、解码渲染与 Demo 标注预览对同一组几何字段的解释方式。

## Goals / Non-Goals

**Goals:**
- 明确 OCR 四边形的标准语义：`poly[0]` 为左上角，后续点按顺时针表示 `topRight`、`bottomRight`、`bottomLeft`
- 基于四边形边长与边方向计算文本块的 `x`、`y`、`width`、`height`、`rotate`，不再以轴对齐包围盒近似
- 让 `ImageParser.encode()`、`ImageParser.decode()` 和 Demo 红框预览共享同一套旋转矩形语义
- 保留异常输入的安全降级路径：`poly` 点数不足、点位重复、边长过短或顺序失真时，仍能回退到稳定输出
- 用规范和测试锁定几何契约，避免后续 OCR 适配或可视化调整再次破坏 `poly` 解析一致性

**Non-Goals:**
- 不更换 OCR 引擎，也不修改第三方 OCR 输出格式
- 不为透视四边形引入新的中间文档字段；仍复用现有 `IntermediateText` 的位置、尺寸和旋转字段
- 不尝试在本次变更中重建任意四边形透视投影；目标是收敛为稳定的旋转矩形语义
- 不改造 Demo 的交互结构，仅修正其几何绘制规则

## Decisions

### 1. 将 `poly` 解析拆分为“有序四边形”与“旋转矩形几何”两层

编码阶段新增统一的 OCR 几何归一化辅助结构：

- 第一层负责把原始 `poly` 校验为有序四边形，优先信任 OCR 输出的原始点序：`[topLeft, topRight, bottomRight, bottomLeft]`
- 第二层基于有序四边形推导旋转矩形语义：
  - `x` / `y` 取左上角点
  - `width` 取上边与下边长度的平均值
  - `height` 取左边与右边长度的平均值
  - `rotate` 取上边与下边方向角的平均值

这样可以把“点序合法性”和“几何指标计算”解耦：前者只处理输入可信度，后者只产出中间文档需要的统一几何字段。

备选方案：
- 继续沿用 `x + y` / `y - x` 的启发式重排：对无序点有一定容错，但会掩盖 OCR 已经提供稳定点序这一事实，并可能在近似菱形或高旋转场景下选错角点
- 继续使用轴对齐包围盒：实现简单，但无法满足本次修复目标

### 2. 几何字段以旋转矩形为准，轴对齐包围盒只保留给采样和兜底

`IntermediateText` 中影响布局的核心字段将统一来自旋转矩形几何，而不是 `poly` 的最小外接轴对齐框：

- `x` / `y`：文本块旋转矩形的左上角锚点
- `width` / `height`：按相对边平均值得到的视觉宽高
- `rotate`：文本主方向角度

但轴对齐包围盒并不会完全移除，而是降级为辅助用途：

- 图片像素采样（例如已有的 `estimateFontWeight()`）继续使用轴对齐框，避免旋转裁剪带来额外复杂度
- 当四边形无效时，仍可退回到当前包围盒路径，保证 `encode()` 不因单个异常块整体失败

这能在不推翻现有像素采样实现的前提下，先修复最关键的几何语义错误。

备选方案：
- 连采样也切换到旋转裁剪：几何更精确，但会额外引入 canvas 旋转裁剪逻辑，超出本次修复的必要范围

### 3. 样式推断与几何推断分离，避免 `skew` 再次污染文本块框体

当前 `inferGeometryStyleHints()` 同时承担方向推断和样式推断，容易让 `skew` 与 `rotate` 混入文本块几何语义。设计上将其拆为两类结果：

- **几何结果**：仅负责 `x`、`y`、`width`、`height`、`rotate`
- **样式结果**：继续负责 `italic`、`skew`、`fontWeight`

`createOcrText()` 组装中间文档时，几何字段只吃几何结果，样式字段只吃样式结果。这样即使后续继续保留斜体或字重启发式，文本块的框体仍然由 `poly` 的旋转矩形语义决定，不会再因为样式推断而改变框体尺寸或贴合方式。

备选方案：
- 继续复用一个 `NormalizedOcrTextBlock` 同时承载所有推断结果：短期改动更少，但职责会继续混杂，后续很难判断某个字段到底属于布局还是样式

### 4. Demo 红框与导出预览共享“旋转矩形”绘制规则

红框预览的目标是校验 OCR 几何，而不是校验字体斜切。因此预览逻辑将与编码语义对齐：

- 红框绘制只使用 `x`、`y`、`width`、`height`、`rotate`
- `skew` 可以继续显示在调试标签里，但不再参与红框路径变换
- 若 `ImageParser.decode()` 内部需要展示或重绘文字，也应基于同一组旋转矩形字段放置文本，确保“中间文档长什么样，预览就怎么贴合”

为避免前后逻辑再次分叉，设计上应抽出共享的几何辅助函数，例如：

- 从 `poly` 生成旋转矩形指标的辅助函数
- 从 `IntermediateText` 生成预览框变换参数的辅助函数

这样 `src/index.ts` 与 `demo/demo.js` 都只消费同一套几何定义，而不是各自再推一遍。

备选方案：
- 保留 Demo 继续对 `skew` 做框体变换：可以更贴近某些斜体效果，但会重新把“样式”和“几何框体”混为一谈，不符合本次修复目标

### 5. 回归测试以“几何契约稳定”而非实现细节为核心

测试与规范的重点不是绑定某个内部 helper，而是锁定外部可观察契约：

- 对标准顺时针四边形，`encode()` 产出的文本块应满足：
  - `x` / `y` 等于左上角点
  - `width` / `height` 接近相对边平均值
  - `rotate` 接近上边方向角
- 对异常 `poly`，输出应稳定回退，不抛出额外错误
- Demo / `decode` 相关预览若消费同一份中间文档，应能以同一套旋转矩形语义绘制框体

这样做可以让后续实现自由调整阈值和 helper 拆分，但不能破坏对外几何语义。

备选方案：
- 只断言某个内部函数返回值：粒度太细，容易让重构成本过高
- 只做人眼检查 Demo：缺乏自动回归保障

## Risks / Trade-offs

- [OCR 在少数场景下返回的点序并非严格左上开始顺时针] → 先校验点序与边关系；校验失败时走兜底重排或轴对齐包围盒回退，不让错误点序污染主流程
- [平均边长后得到的旋转矩形无法完全贴合强透视四边形] → 明确本次只保证旋转矩形语义；对明显透视变形仍允许存在有限误差，但保持一致、可预测
- [预览去掉 `skew` 后，部分斜体文字的红框主观上看起来不如之前“贴边”] → 把红框定位为几何校验层，把 `skew` 留在样式标签和文字渲染中，避免职责混淆
- [旧测试依赖轴对齐包围盒宽高，修改后可能出现断言批量变化] → 只更新受该契约影响的断言，并补上新几何样本，避免用快照掩盖语义变化

## Migration Plan

1. 在 `src/index.ts` 中引入新的 OCR 四边形归一化 helper，明确原始点序校验与兜底策略。
2. 将 `normalizeOcrResult()` 改为先产出旋转矩形几何，再分别叠加样式提示和像素采样结果。
3. 更新 `createOcrText()` 的输入结构，使 `width`、`height`、`x`、`y`、`rotate` 全部来自新的几何结果。
4. 调整 `demo/demo.js` 的红框绘制逻辑，只按旋转矩形语义渲染标注框，并继续保留样式标签展示。
5. 补充 `src/__tests__/imageParser.test.ts` 中的几何回归样例，覆盖标准旋转框、轻微梯形、异常点序 / 非法 `poly` 回退等场景。
6. 视需要同步更新对应 spec，明确 `poly` 点序契约和输出几何字段含义。

回滚策略：若上线后发现某些 OCR 运行时存在大量非标准点序，可保留新 helper，但临时让其在校验失败时统一回退到旧包围盒路径，以确保基础 OCR 输出稳定。

## Open Questions

- 对“点序合法但边不完全平行”的轻微梯形，是否允许继续输出平均宽高 + 主方向角，还是需要设置更严格阈值直接回退？
- Demo 红框是否需要额外支持显示原始 `poly` 轮廓，作为旋转矩形之外的调试对照层？
- `decode()` 若继续使用 `skew` 渲染文字，是否需要单独补一条规范，明确“文字样式变换不影响几何框体语义”？
