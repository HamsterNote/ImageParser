## Context

当前 `ImageParser.decode()` 会在 `drawDecodedPage()` 中为每个 `IntermediateText` 设置字体、颜色、方向、旋转与斜切，然后直接调用：

- `context.fillText(text.content, 0, 0, Math.max(1, text.width))`

这条路径只能把文字绘制在“最多不超过 `text.width`”的宽度内，但不能保证最终渲染宽度与 `IntermediateText.width` 严格一致：

- 当字体自然宽度大于 `text.width` 时，浏览器可能压缩到不超出上限
- 当字体自然宽度小于 `text.width` 时，Canvas 不会主动把文字拉伸到目标宽度
- 当前实现也没有显式测量实际字宽，因此无法判断本次绘制与中间文档几何宽度之间的偏差

这使得 `ImageParser.decode()` 虽然能把文字回放到正确位置，但在宽度维度上仍可能与 `IntermediateText.width` 不一致，导致解码预览与中间文档中的文本框语义脱节。

本次变更只收紧解码阶段的绘制契约：在不修改 `ImageParser` 输入输出形式、不新增外部依赖的前提下，让解码得到的文字回放宽度与当前文本框宽度对齐，并保持现有旋转、斜切、方向与导出流程稳定。

## Goals / Non-Goals

**Goals:**
- 让 `ImageParser.decode()` 中每段文本的最终绘制宽度与 `IntermediateText.width` 对齐，而不是仅依赖 Canvas 默认字宽
- 保持 `IntermediateText` 现有字段语义不变，继续复用 `x`、`y`、`width`、`height`、`rotate`、`skew`、`dir` 等信息完成解码
- 将宽度适配限制在解码绘制链路内，避免反向改动编码阶段或中间文档结构
- 为空文本、非法宽度、不可测量文本等边界情况提供安全降级，避免解码过程引入新的异常
- 为后续 spec delta 和测试补充明确的“宽度对齐”约束，防止该行为再次回退为“仅限制最大宽度”

**Non-Goals:**
- 不修改 `ImageParser.encode()` 的输出结构，也不重新推导 OCR 几何字段
- 不引入新的字体排版引擎、字偶距算法或复杂文本布局系统
- 不试图让回放文字在字形、换行、字距上与原图逐像素完全一致；本次只保证宽度契约更严格
- 不调整 Demo 交互层或公开 API，仅修正 `decode()` 的文字回放行为

## Decisions

### 1. 在解码阶段新增“先测量，再按目标宽度缩放”的文本宽度适配步骤

`drawDecodedPage()` 将从“直接调用 `fillText(..., maxWidth)`”升级为两段式渲染：

1. 使用与当前文本一致的 `font`、`direction` 与上下文设置先测量自然字宽
2. 基于 `targetWidth = text.width` 与 `measuredWidth` 计算横向缩放比，并在本地坐标系下应用后再绘制文本

核心原因是 Canvas 的 `maxWidth` 参数只提供“不要超过”的约束，不提供“必须等于”的契约。要让最终渲染宽度与 `IntermediateText.width` 严格一致，必须显式知道自然字宽，并将差值转化为可控的变换。

设计上优先复用 Canvas 原生测量能力，不引入额外依赖，也不依赖浏览器对 `maxWidth` 的隐式实现差异。

备选方案：
- 继续只使用 `fillText(..., maxWidth)`：实现最简单，但无法满足“宽度必须对齐”的要求
- 通过调整 `fontSize` 反复逼近目标宽度：会同时改变字高、基线与行内视觉比例，破坏现有几何语义
- 通过 `letterSpacing` 或逐字布局拉伸：Canvas 2D 原生支持不一致，实现复杂度高，且超出本次需求

### 2. 宽度适配只作用于文本本地 X 轴，保持锚点、基线与垂直尺寸稳定

新的宽度适配不会改动 `x`、`baselineY`、`rotate`、`skew` 的既有求值方式，而是在完成 `translate`、`rotate`、`skew` 后，再对文本本地坐标系应用横向缩放。这样可以保证：

- 文本仍以当前左上角 / 基线锚点进入绘制流程
- 旋转文字继续围绕相同锚点放置，不会因为世界坐标缩放而偏移
- 文字高度、基线与现有 `fontSize` 语义保持一致，只修正宽度维度

这比直接修改世界坐标或整体缩放 `canvas` 更安全，也更符合 `IntermediateText.width` 是“文本行内宽度约束”的语义。

备选方案：
- 在 `translate` 之前对上下文做缩放：会把位置偏移也一并放大，导致文本锚点漂移
- 同时缩放 X/Y 轴：虽然实现更直接，但会破坏文字高度与当前 `fontSize`、`height` 的关系

### 3. 将宽度测量与缩放计算封装为独立 helper，解耦渲染流程与边界处理

为避免 `drawDecodedPage()` 内部继续堆积条件分支，设计上新增专门的解码宽度适配 helper（例如“测量文本自然宽度”和“计算安全缩放比”两层能力），职责划分如下：

- **测量层**：负责基于当前 Canvas 上下文获取可用的自然字宽
- **适配层**：负责把 `targetWidth` 与 `measuredWidth` 转成最终缩放策略，并处理非法输入
- **渲染层**：只消费计算结果并执行 `save/restore`、变换和 `fillText`

这样可以把“怎么测”、“什么时候降级”、“怎么画”拆开，便于在测试中直接锁定宽度适配契约，而不是把所有行为揉进一个大函数里。

备选方案：
- 把测量、判断和绘制都塞回 `drawDecodedPage()`：短期改动更少，但后续很难单独验证不同边界场景

### 4. 对不可安全适配的输入显式降级，而不是让宽度修复变成新的失败源

为了保证 `decode()` 的鲁棒性，新逻辑只在以下条件满足时启用精确宽度适配：

- `text.content` 非空且存在可测量字宽
- `text.width` 为有限且大于 0 的目标宽度
- Canvas 上下文支持稳定的文本测量结果

若任一条件不满足，则降级为当前稳定路径继续绘制文本，而不是抛出新的错误。这样可以让宽度契约在“正常文本块”上收紧，同时保持异常内容仍可导出图片。

对于仅包含空白、测量结果为 0、或字体回退导致结果异常的场景，系统仍优先保证解码成功；这些场景可在规范里明确为“允许降级，不承诺精确拉伸”。

备选方案：
- 只要无法精确适配就直接抛错：会把视觉增强变成稳定性回退，不符合解码容错目标
- 对所有异常测量结果强行使用经验缩放比：表面上覆盖更多场景，但会制造更难诊断的失真输出

### 5. 通过 spec delta 与测试把“最大宽度”升级为“目标宽度契约”

本次设计不把“宽度对齐”停留在实现细节，而是会同步沉淀到变更规范中：

- 在 `image-ocr-decode-overlay-preview` 的 delta spec 中新增 `ImageParser.decode()` 文本回放宽度必须与 `IntermediateText.width` 对齐的约束
- 在实现测试中补充可观察断言，验证宽度适配 helper 或绘制流程对给定目标宽度产生一致结果

测试重点应放在外部契约，而不是某个具体 helper 名称。允许内部重构，但不允许语义再次退回成“仅限制最大宽度”。

备选方案：
- 只改代码不补 spec / 测试：短期成本低，但后续非常容易被浏览器兼容或重构再次破坏

## Risks / Trade-offs

- [不同浏览器对 `measureText()` 与字体回退的实现略有差异] → 统一使用当前实际渲染字体进行测量，并在规范与测试中接受合理的像素级容差，而不是要求逐像素完全相同
- [横向缩放会让部分字体在极端比例下出现主观失真] → 仅对已存在的 `IntermediateText.width` 契约做对齐，不改变字高与基线；异常比例场景继续允许安全降级
- [空文本或空白文本难以得到稳定自然宽度] → 将其明确归入降级路径，优先保证解码稳定而非强求精确拉伸
- [新增 helper 后若职责划分不清，后续可能再次把测量、容错与绘制耦合在一起] → 通过设计约束 helper 分层，并让测试直接覆盖“测量/适配/渲染”边界

## Migration Plan

1. 在 `src/index.ts` 中为解码绘制新增文本自然宽度测量与目标宽度适配 helper。
2. 调整 `drawDecodedPage()` 的绘制顺序：保留现有字体、方向、旋转与斜切逻辑，在本地文本坐标系中追加横向宽度适配。
3. 为不可测量、空内容或非法目标宽度场景补充安全降级分支，确保 `decode()` 仍可导出图片。
4. 在当前变更下补充对应 spec delta，明确“解码文本回放宽度与 `IntermediateText.width` 对齐”的行为要求。
5. 在测试中增加与该契约直接相关的回归场景，覆盖正常适配与降级路径。

回滚策略：若新逻辑在某些字体环境下出现不可接受的显示失真，可保留 helper 结构，但暂时关闭横向缩放应用，回退到旧的 `fillText(..., maxWidth)` 路径，同时保留规范与测试入口以便后续重新启用。

## Open Questions

- 对只包含空格或制表符的文本块，规范是否需要明确为“允许跳过精确宽度对齐，仅保证可导出”？
- 测试应主要锁定 helper 的缩放比计算，还是补充基于 Canvas mock 的端到端绘制断言，以更贴近真实行为？
- 如果未来 Demo 也直接展示 `decode()` 的宽度适配结果，是否需要补一条规范说明“回放文本宽度契约优先于字体自然字宽”？
